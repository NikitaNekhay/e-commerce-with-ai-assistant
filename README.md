# AI-ассистент для улучшения объявлений маркетплейса

Веб-приложение — личный кабинет продавца с интегрированным AI-ассистентом, который помогает улучшать описания объявлений и оценивать рыночную стоимость товаров.

## Технологический стек

### Frontend
- **React 18** + TypeScript
- **React Router v7** — маршрутизация
- **Redux Toolkit** — управление состоянием
- **Ant Design v6** — UI-компоненты
- **Tailwind CSS v4** — стилизация
- **Axios** — HTTP-запросы
- **date-fns** — форматирование дат
- **Vite** — сборка

### Backend
- **Express.js** + TypeScript
- **OpenRouter API** — интеграция с LLM (Google Gemini 2.0 Flash по умолчанию)

### Инфраструктура
- **Docker** + **docker-compose** — контейнеризация
- **Nginx** — раздача статики и проксирование API
- **Vitest** + **Testing Library** — юнит-тесты

###  Docker Compose

## Реализованный функционал

### Обязательные требования

1. **Страница списка объявлений** (`/ads`)
   - Серверный поиск по названию с debounce 300мс
   - Сортировка: по названию (А→Я, Я→А), новизне, цене
   - Мультикатегорийный фильтр (чекбоксы)
   - Фильтр «Только требующие доработок»
   - Серверная пагинация (10 на страницу)
   - Переключатель сетка/список
   - Кнопка сброса фильтров

2. **Страница просмотра** (`/ads/:id`)
   - Полная информация: название, цена, категория, дата публикации/редактирования
   - Характеристики зависят от категории (показываются только заполненные)
   - Баннер «Требуются доработки» с перечнем пустых полей (вычисляется клиентски)
   - Описание: текст или «Отсутствует»

3. **Страница редактирования** (`/ads/:id/edit`)
   - Категория влияет на набор характеристик:
     - Электроника: тип, бренд, модель, состояние, цвет
     - Транспорт: бренд, модель, год, КПП, пробег, мощность
     - Недвижимость: тип, адрес, площадь, этаж
   - AI-функции: «Придумать/Улучшить описание», «Узнать рыночную цену»
   - Сохранение черновика в localStorage с предложением восстановить
   - Валидация обязательных полей

4. **Серверная часть** — Express.js API с эндпоинтами GET /items, GET /items/:id, PUT /items/:id

5. **LLM-интеграция** — OpenRouter API (серверный прокси, ключ на бэкенде)

### Дополнительные требования (со звёздочкой)

1. **Docker Compose** — клиент (nginx) + сервер запускаются одной командой
2. **Безопасность LLM** — правила в `server/src/prompts/safety.md`, защита от prompt injection
3. **Прерывание запросов** — AbortController при навигации между страницами (в Redux thunks и AI-запросах)
4. **Юнит-тесты** — Vitest + Testing Library (24 теста: утилиты, Redux slices)
5. **Чат с AI** — панель чата на странице редактирования с контекстом объявления
6. **Визуальное сравнение «Было → Стало»** — word-level diff при генерации описания
7. **Тёмная тема** — переключатель в шапке, сохранение в localStorage, Ant Design darkAlgorithm

## Принятые решения

1. **Сортировка по цене**: API-контракт указывает `sortColumn: "title" | "createdAt"`, но ТЗ требует сортировку по цене. Добавлена поддержка `sortColumn: "price"` на сервере.

2. **Вычисление needsRevision на клиенте**: Согласно ТЗ, параметр вычисляется клиентской стороной. Объявление считается требующим доработок, если не заполнено описание или любой из параметров категории.

3. **OpenRouter вместо Ollama/Grok**: Выбран OpenRouter как наиболее гибкий вариант — единый API для множества моделей. По умолчанию используется Gemini 2.0 Flash (быстрая и бесплатная модель).

4. **Серверный прокси для LLM**: API-ключ хранится только на сервере в `.env`, фронтенд обращается через прокси `/api/ai/*`. Приватные токены не попадают в клиентский код.

5. **Word-level diff без внешних зависимостей**: Реализован собственный LCS-алгоритм для сравнения текстов — минимизация зависимостей.

6. **Мультикатегорийный фильтр**: ТЗ упоминает фильтрацию «по категории/категориям» — реализовано через чекбоксы с возможностью выбора нескольких категорий одновременно.
